Полное руководство по проектам GitHub

Что такое проекты GitHub?

Проекты в GitHub — это адаптируемый инструмент для планирования и отслеживания работы над программным обеспечением или любыми другими задачами. Они интегрируются с issues и pull requests, позволяя организовать работу на уровне репозитория, организации или пользователя. Проекты предоставляют три основных вида отображения:

  • Доска (Board): Визуализация задач в стиле Kanban с колонками, например, "To do", "In progress", "Done".
  • Таблица (Table): Табличное представление с настраиваемыми полями (статус, приоритет, дедлайн).
  • Временная шкала (Roadmap): Отображение задач по датам для планирования этапов.

Проекты поддерживают автоматизацию (например, перемещение задач при изменении статуса) и совместную работу в реальном времени. Они заменили устаревшие "Projects (classic)" и стали основным инструментом управления задачами в GitHub.

Пример использования

Команда разработчиков создает проект "Release v1.0". На доске они размещают задачи: "Исправить баг #45" (In progress), "Добавить README" (To do), а на временной шкале отмечают дедлайн релиза — 30 марта 2025.

Создание нового проекта

Создание проекта начинается с выбора уровня: пользователь (personal), репозиторий или организация. Вот пошаговый процесс:

  1. Перейдите во вкладку "Projects" (на уровне профиля, репозитория или организации).
  2. Нажмите "New project" или зеленую кнопку "Create new project".
  3. Выберите шаблон:
    • Basic Kanban: Простая доска с колонками To do, In progress, Done.
    • Bug triage: Для сортировки багов с полями Priority и Severity.
    • Team planning: Для планирования командной работы с датами.
    • Blank project: Пустой проект для полной настройки.
  4. Настройте поля: добавьте статус, приоритет, дедлайн, ответственных и т.д.
  5. Сохраните проект и начните добавлять задачи (issues, pull requests или вручную).

Проекты можно сделать приватными (только для владельца/организации) или публичными. Также доступны настройки доступа для коллабораторов.

Пример создания

В организации "MyOrg" администратор переходит в "Projects", выбирает "New project", использует шаблон "Team planning", добавляет поле "Due date" (например, 15 апреля 2025) и задачу "Обновить API" с приоритетом "High".

Копирование существующего проекта

Функция копирования позволяет быстро создать новый проект на основе существующего, сохраняя структуру и настройки. Процесс:

  1. Откройте проект и нажмите на меню настроек (⋮).
  2. Выберите "Copy".
  3. Укажите параметры копирования:
    • Items: Копировать задачи (issues/pull requests).
    • Draft issues: Включить черновики задач.
    • Fields: Сохранить пользовательские поля (например, Priority).
    • Views: Перенести виды (доска, таблица).
  4. Выберите новое местоположение (репозиторий, организация или личный профиль).
  5. Нажмите "Copy project".

Копия создается как независимый проект, который можно дальше редактировать.

Пример копирования

Проект "Sprint 1" с доской (To do, In progress, Done) и полем "Story points" копируется для "Sprint 2". Пользователь выбирает копирование только структуры без задач, чтобы начать новый спринт с чистого листа.

Миграция из Projects (classic)

GitHub Projects (classic) устарели, и пользователям рекомендуется перейти на новые проекты. Для миграции:

  1. Откройте классический проект.
  2. Нажмите "Menu" → "Export project data".
  3. Скачайте файл JSON с данными проекта.
  4. Создайте новый проект в разделе "Projects".
  5. В новом проекте выберите "Import from classic" и загрузите JSON-файл.

После импорта проверьте данные: некоторые функции (например, старые автоматизации) могут не поддерживаться. Новый проект предлагает улучшенные возможности, такие как временные шкалы и настраиваемые поля.

Пример миграции

Пользователь экспортирует классический проект "Old Roadmap" в JSON, затем в новом проекте выбирает "Import from classic", загружает файл и получает доску с задачами вроде "Fix UI" и обновляет дедлайны.
